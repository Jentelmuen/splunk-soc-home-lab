\# 🔍 Модуль 2: Емуляція та аналіз атак



\## 🛠️ Кейс із практики: Вирішення проблеми блокування логів (Troubleshooting)



Під час першого запуску служби `SplunkForwarder` через PowerShell лог-файли не з'явилися в пошуку, а у внутрішніх логах форвардера (`splunkd.log`) було зафіксовано помилку:

`Connect to 127.0.0.1:9997 failed. Подключение не установлено, т.к. конечный компьютер отверг запрос на подключение.`



\### Хронологія розслідування та вирішення:

1\. \*\*Аналіз прав доступу:\*\* Спробу керування службою було заблоковано системою Windows (`Access is denied`). Інструмент \*\*Process Explorer\*\* та консоль \*\*PowerShell\*\* були перезапущені з підвищеними привілеями (Адміністратор) для успішного рестарту служби: `Restart-Service -Name SplunkForwarder`.

2\. \*\*Локалізація проблеми (Мережевий рівень):\*\* Порт `9997` у віртуальній машині виявився закритим внутрішнім фаєрволом Ubuntu (UFW), який блокував вхідний трафік від VirtualBox.

3\. \*\*Вирішення:\*\* У терміналі Linux було виконано команду для відкриття порту:

&#x20;  ```bash

&#x20;  sudo ufw allow 9997/tcp

Після відкриття порту на фаєрволі та перезапуску служби на Windows, зв'язок успішно встановився, і логи безпеки Windows (Event Logs) почали стабільно індексуватися в SIEM.



!\[Результати успішного збору логів хостової системи у Splunk](img/windows\_logs-success.png)



🔍 Емуляція та аналіз атаки: Brute-Force (MITRE ATT\&CK T1110)

Щоб перевірити ефективність налаштованого збору логів, було проведено симуляцію атаки методом перебору паролів (Brute-Force) на локальні облікові записи Windows за допомогою фреймворку Atomic Red Team (техніка T1110).



1\. Емуляція атаки (Red Team Action)

Через PowerShell від імені Адміністратора було запущено атомарний тест, який ініціював циклічні спроби автентифікації до локальних ресурсів системи з використанням фреймворку автоматизації:





Для генерації масового потоку подій було використано циклічний PowerShell-скрипт, що імітує підбір паролів до мережевої папки за 10 секунд:



for ($i=1; $i -le 50; $i++) {

&#x20;   net use \\\\127.0.0.1\\c$ /user:TargetAdmin WrongPassWord123! 2>$null

}



У результаті атаки Windows Security Log згенерував масив критичних подій невдалого входу. Для аналізу аномалії в Splunk було розроблено аналітичний запит на мові SPL (Splunk Processing Language).



SPL-запит для виявлення брутфорсу:



index="windows\_security" EventCode=4625



🔬 Анатомія події (Аналіз IoC / Маркерів компрометації):

При глибокому аналізі сирого логу (Raw Event) у Splunk було виділено такі ключові артефакти:



EventCode = 4625 — у системі Windows цей ідентифікатор відповідає виключно за невдалу спробу входу (An account failed to log on).



TargetUserName = TargetAdmin — чітко вказує на обліковий запис, який намагалися зламати.

!\[Інформація про подію (eventcode)](img/log\_information1.png)

IpAddress (Source Network Address) = 127.0.0.1 — джерело атаки (у межах лабораторної роботи — локальний хост).



Logon Type = 3 — мережевий тип входу (Network Logon). Свідчить про те, що спроба підбору здійснювалася через мережеві протоколи, а не локально на клавіатурі.



Status / Sub Status = 0xC000006A — найважливіший маркер. Цей специфічний код помилки Windows розшифровується як: "Ім'я користувача вказано вірно, але введений пароль — помилковий". Наявність десятків таких подій із суб-статусом 0xC000006A за короткий проміжок часу є 100% підтвердженням Brute-Force атаки, а не випадкової помилки користувача.

🛡️ Стратегія захисту та протидії (Mitigation)

На основі якого розслідування інциденту, для захисту інфраструктури розроблено наступний комплекс заходів:



Політика блокування облікових записів (Account Lockout Policy): Налаштування групових політик (GPO) Windows для автоматичного тимчасового блокування облікового запису (наприклад, на 15-30 хвилин) після 5 послідовних невдалих спроб введення пароля протягом 5 хвилин. Це робить автоматизований брутфорс технічно неефективним.



Багатофакторна автентифікація (MFA): Впровадження обов'язкового другого фактора для всіх критичних облікових записів та віддалених підключень (VPN, RDP).



SIEM Alerting (Автоматичне реагування): Трансляція розробленого SPL-запиту в постійно діюче правило сповіщення (Alert). У разі фіксації понад 20 спроб невдалого входу з одного IP протягом 1 хвилини, SIEM автоматично генерує інцидент High-пріоритету для чергового SOC-аналітика, що дозволяє миттєво ізолювати атакуючий хост на рівні мережі.
